## 📝 Конспект

### Проблема качества в мире агентов

Агенты по своей природе **недетерминированы** — один и тот же запрос может дать разные результаты. Это ломает традиционные подходы к QA.

> **Принцип:** Качество агента — это **архитектурный столп**, а не финальная фаза тестирования.

### Три столпа качества

```
┌──────────────────────────────────────────────────┐
│              КАЧЕСТВО АГЕНТА                      │
├──────────────┬────────────────┬───────────────────┤
│  Логирование │   Трассировка  │    Метрики        │
│  (Logging)   │   (Tracing)    │    (Metrics)      │
├──────────────┼────────────────┼───────────────────┤
│ Что произошло│ Как это        │ Насколько хорошо  │
│              │ произошло      │ это работает      │
│ Журнал       │ Цепочка        │ Latency, tokens,  │
│ событий      │ вызовов        │ success rate      │
└──────────────┴────────────────┴───────────────────┘
```

### Оценка агентов (Evaluation)

**Фреймворки оценки:**
1. **Точность ответа** — насколько релевантен и правилен ответ
2. **Использование инструментов** — правильно ли агент выбирает и вызывает инструменты
3. **Следование инструкциям** — соблюдает ли агент свои правила
4. **Устойчивость** — стабильность при повторных запусках

### LLM как судья (LLM-as-a-Judge)

Использование другой LLM для оценки качества ответов агента:
- Быстрее и дешевле ручной оценки
- Масштабируется на тысячи тестов
- Требует калибровки и бенчмарков

### Наблюдаемость (Observability)

| Компонент | Что отслеживает | Инструменты |
|-----------|----------------|-------------|
| **Логи** | События, ошибки, предупреждения | Cloud Logging |
| **Трейсы** | Полный путь запроса через систему | OpenTelemetry |
| **Метрики** | Производительность, стоимость, качество | Prometheus, Grafana |

## 🔧 Пошаговая инструкция: Наблюдаемость агентов

### Шаг 1: Включение логирования

```python
import logging

# Настройка подробного логирования ADK
logging.basicConfig(level=logging.INFO)
logger = logging.getLogger("agent_quality")
```

### Шаг 2: Использование трассировки

```python
# ADK поддерживает OpenTelemetry из коробки
# Трейсы показывают полную цепочку: запрос → рассуждение → вызов инструмента → ответ

from google.adk.runners import InMemoryRunner

runner = InMemoryRunner(agent=root_agent)

# run_debug автоматически выводит пошаговую трассировку
response = await runner.run_debug("Какой курс доллара сегодня?")
```

### Шаг 3: Создание оценочных тестов

```python
# Набор тестов для оценки качества агента
test_cases = [
    {
        "input": "Конвертируй 100 USD в EUR",
        "expected_tools": ["get_exchange_rate"],
        "expected_contains": ["EUR", "93"]
    },
    {
        "input": "Что такое MCP?",
        "expected_tools": ["google_search"],
        "expected_contains": ["Model Context Protocol"]
    }
]

# Прогон тестов
for test in test_cases:
    response = await runner.run_debug(test["input"])
    # Проверяем, содержит ли ответ ожидаемые элементы
    for keyword in test["expected_contains"]:
        assert keyword in str(response), f"Не найдено: {keyword}"
```

### Шаг 4: LLM-as-a-Judge

```python
# Использование другой LLM для оценки ответов
judge_prompt = """Оцени качество ответа агента по шкале 1-5:

Вопрос: {question}
Ответ агента: {response}

Критерии:
1 - Полностью неправильный
2 - Есть ошибки
3 - Приемлемо
4 - Хорошо
5 - Отлично

Оценка:"""
```

### Шаг 5: Мониторинг в продакшене

```python
# Ключевые метрики для отслеживания
metrics_to_track = {
```

## 💡 Ключевые выводы

1. **Качество — это архитектура**, а не тестирование «в конце»
2. **Три столпа:** логирование, трассировка, метрики
3. **LLM-as-a-Judge** — масштабируемый способ оценки
4. **Недетерминизм** — главная проблема, требующая новых подходов к тестированию
5. ADK Web UI — самый удобный способ **визуально отладить** агента
